<h2>Overview</h2>
<p>Note: This writeup and the challenge binary are available <a href="https://github.com/kernel-sanders/nsec2020-hack-the-time" target="_blank">on GitHub</a>.</p>
<p>This is a walk-through of "Hack The Time" a 4-point challenge from the 2020 NSEC CTF. It was a great challenge that required static analysis, dynamic analysis, web skills, Go knowledge, and some creative Bash foo to solve. This was team effort with help from two of my teammates (finding the arg and some bash foo).</p>
<p>Props to <a href="https://twitter.com/becojo" target="_blank">@becojo</a> for a great challenge!</p>
<h2>Challenge</h2>
<div class="highlight"><pre><span></span><code>Classes are sooooooo long and I’m bored af. There is a rumour that the student that developped the service that controls
the school’s clock added a backdoor to change the time. It is still running to this day… It would be really cool if 
you can find it :wink:

I was able to get a copy of the binary if that helps you: https://dl.nsec/time_server [note binary included in this repo]

The school server is running at http://time-server.ctf:8080.
</code></pre></div>

<h3><em>Static Analysis</em></h3>
<hr/>
<p>Opening the binary in IDA Pro we see lots of <code>go_</code> and <code>net_http__</code> functions.</p>
<p><img alt="main.main" src="images/main.main.png"/></p>
<p>So we have a Go binary. It's a mess, but <a href="https://github.com/strazzere/golang_loader_assist" target="_blank">golang_loader_assist</a> helps resolve function names and strings, and generally makes it a little more pleasant to work with.</p>
<p>You can see in the screenshot above of <code>main.main</code> there are two handlers being registered, <code>/time.json</code> and <code>/</code>, which makes sense given what we saw on the challenge site. There are no other handlers, so the backdoor isn't as easy as <code>/cmd</code> or similar.</p>
<p>Let's look at the <code>/time.json</code> handler (<code>main_timeHandler</code>)</p>
<p><img alt="timeHandler" src="images/timeHandler.png"/></p>
<p>It gets the current time (<code>time_Now</code>), formats it (<code>time_Time_Format</code>), coverts it to a string (<code>runtime_convTstring</code>), and finally prints it (<code>fmt_Fprintf</code>). Nothing jumps out as being a backdoor, so let's move on for now. We can come back and reverse engineer this function in more depth if we don't find anything else.</p>
<p>How about <code>main_root</code> the handler for <code>/</code>?</p>
<p><img alt="main_root" src="images/main_root.png"/></p>
<p>Nothing super interesting. Follow the rabbit hole down <code>main_root_func1</code>.</p>
<p><img alt="main_root_func1" src="images/main_root_func1.png"/></p>
<p>Keep going into <code>main_e</code></p>
<p><img alt="main_e" src="images/main_e.png"/></p>
<p>Well well well. <code>net_http__Request_FormValue</code> with a strange string <code>mlaasdkfasldkfm</code>. This feels like a backdoor. How can we send that string to have it read by <code>FormValue</code>? To the <a href="https://golang.org/src/net/http/request.go?h=FormValue" target="_blank">docs</a>!</p>
<div class="highlight"><pre><span></span><code><span class="c1">// FormValue returns the first value for the named component of the query.</span>
<span class="c1">// POST and PUT body parameters take precedence over URL query string values.</span>
<span class="c1">// FormValue calls ParseMultipartForm and ParseForm if necessary and ignores</span>
<span class="c1">// any errors returned by these functions.</span>
<span class="c1">// If key is not present, FormValue returns the empty string.</span>
<span class="c1">// To access multiple values of the same key, call ParseForm and</span>
<span class="c1">// then inspect Request.Form directly.</span>
</code></pre></div>
<p>Since we know that <code>/</code> works with <code>GET</code>, we must have to send that string as "name component of the query" aka a GET parameter. The string will be the "key" and whatever it's value is will be returned by <code>FormValue</code>. Something like:</p>
<div class="highlight"><pre><span></span><code>curl<span class="w"> </span>-v<span class="w"> </span>http://127.0.0.1:8080/?mlaasdkfasldkfm<span class="o">=</span><span class="nb">test</span>
</code></pre></div>
<p>Ok so that should get us to the <code>jbe short loc_76320C</code> test. What is that checking? Since this is a 64bit ELF, we can see if IDA's pseudocode helps any:</p>
<p><img alt="main_e_code" src="images/main_e_code.png"/></p>
<p>If something is &lt;=1 it will panic, but we're not quite sure what that something is. Looking further to line 17, there are two checks against something related to <code>os_Args</code> (<code>os_Args + 24</code> and <code>os_Args +16</code>). If all three of these checks pass, the program will call <code>main_rr</code>. Let's look at <code>main_rr</code>, with a mental note to come back and figure out what these checks are for.</p>
<p><img alt="main_rr" src="images/main_rr.png"/></p>
<p>As soon as we see <code>os_exec_Command</code>:</p>
<p><img alt="there_it_is" src="images/there_it_is.gif"/></p>
<p>At this point we're confident we have found the backdoor, but it looks like it has additional checks and it isn't obvious what those checks are for. It's time for dynamic analysis.</p>

<p>After testing, I had the following list: <code>={}[]&lt;&gt;():;"\/</code> (a space is the last character in that list). That's pretty restrictive, as most everything we would want to do in bash would involve a slash and a space.</p>
<p>As a CTF participant and security researcher I've escaped my way out of a fair number of character restricted command injections, but this one really upped the difficulty. A common trick to get around space restriction is with input/output redirection (<code>&lt;</code> and <code>&gt;</code>) but those are restricted as well. <code>${IFS}</code> is another favorite, but curly braces are restricted. We better hope the flag is in the current directory as <code>/</code> is restricted.</p>
